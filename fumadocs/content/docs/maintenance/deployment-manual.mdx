# Deploying SEMS2 to Production

This guide provides comprehensive instructions for deploying the Student Experience Management System (SEMS2) on Dokploy with a self-hosted Supabase instance.

## Overview

SEMS2 is a monolithic SvelteKit application that manages events, registrations, and student organizations. The deployment architecture consists of:

- **Frontend & Backend**: SvelteKit 2.16 monolithic application with server-side rendering and API routes
- **Database**: PostgreSQL 17 via self-hosted Supabase
- **Email Service**: Resend for transactional email delivery
- **Authentication**: Supabase Auth with Google OAuth integration
- **Deployment Platform**: Dokploy PaaS (Platform as a Service)

### Environment Variables Flow

Environment variables are injected at different stages of the deployment lifecycle:

- **Build-time** (`PUBLIC_*`): Embedded into the Docker image during the build process
  - `PUBLIC_SUPABASE_URL`
  - `PUBLIC_SUPABASE_ANON_KEY`
- **Runtime** (Server-only): Injected when the container starts
  - `SUPABASE_SERVICE_ROLE_KEY`
  - `RESEND_API_KEY`
  - `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`

## Prerequisites

Before beginning the deployment process, ensure the following requirements are met:

### Required Access & Accounts

- **Linux VPS or Dedicated Server**: A server running Ubuntu LTS with Dokploy PaaS installed
- **Dokploy Account**: Administrative access to the Dokploy server dashboard
- **Domain Name**: A registered domain for the SEMS2 deployment (e.g., `sems2.university.edu`)
- **Git Repository**: Access credentials for the SEMS2 repository
- **Resend Account**: API access for transactional email delivery ([resend.com](https://resend.com))
- **Google Cloud Console**: OAuth 2.0 credentials for authentication ([console.cloud.google.com](https://console.cloud.google.com))

### Local Development Tools

- **Docker**: Required for running the Supabase CLI
- **Supabase CLI**: For executing database migrations
  ```bash
  npm install -g supabase
  ```
- **Node.js 20+**: Required for local build testing (optional)
- **Git**: For repository management

### Technical Knowledge

- Basic understanding of Docker containerization
- Familiarity with environment variable configuration
- Basic PostgreSQL/SQL knowledge for database migrations
- Understanding of DNS record management

## Deployment Steps

<Step>
### Provision a Cloud Server

Deploy an Ubuntu LTS server from a cloud infrastructure provider. The instructions below reference DigitalOcean, which was used during the initial SEMS2 prototype development. However, any cloud provider that supports Ubuntu LTS deployment is suitable.

1. **Create a new server instance**:

   - Operating System: Ubuntu 24.04 LTS
   - Minimum Specifications: 8GB RAM, 4 vCPUs (required for Docker image builds via Dokploy)
   - Security: Configure SSH keys for secure access
   - Initialize and deploy the server instance

2. **Update the system**:

   ```bash
   sudo apt update && sudo apt full-upgrade -y
   ```

3. **Implement security hardening**:

   Configure essential security measures including:
   - UFW (Uncomplicated Firewall) configuration
   - SSH key-only authentication
   - Root login disabled
   - Fail2ban installation (recommended)

   For comprehensive security guidance, refer to the [Ubuntu Security Hardening Guide](https://ubuntu.com/blog/what-is-system-hardening-definition-and-best-practices).

</Step>

<Step>
### Install Dokploy PaaS

Install Dokploy on the provisioned server by following the official [Dokploy Installation Guide](https://docs.dokploy.com/docs/core/installation). The installation process configures Docker, networking, and the Dokploy dashboard.

</Step>

### Deploy Supabase Instance

Deploy a self-hosted Supabase instance using the official Dokploy Supabase template.

1. **Create a new Dokploy project**:

   - Navigate to the Dokploy dashboard
   - Create a new project (e.g., "SEMS2 Production")

2. **Initialize Supabase service**:

   - Within the project, select **Create Service** → **Template**
   - Filter templates by "Supabase"
   - Select the Supabase template (version 1.25.04, Dokploy ≥ 0.22.5)

3. **Configure Google OAuth integration**:

   Modify the `docker-compose.yml` file to enable Google OAuth. Locate the `auth` service and add the following environment variables:

   ```yaml
   auth:
     environment:
       # Enable External Google Auth
       GOTRUE_EXTERNAL_GOOGLE_ENABLED: ${EXTERNAL_GOOGLE_AUTH_ENABLED}
       GOTRUE_EXTERNAL_GOOGLE_CLIENT_ID: ${GOOGLE_CLIENT_ID}
       GOTRUE_EXTERNAL_GOOGLE_SECRET: ${GOOGLE_CLIENT_SECRET}
       GOTRUE_EXTERNAL_GOOGLE_REDIRECT_URI: ${EXTERNAL_GOOGLE_REDIRECT_URI}
   ```

### Execute Database Migrations

Apply the SEMS2 database schema to the Supabase instance using the Supabase CLI.

1. **Navigate to the project root** on the local development machine:

   ```bash
   cd sems2-prototype
   ```

2. **Configure SSL mode** for the database connection:

   **Windows (PowerShell)**:
   ```powershell
   $env:PGSSLMODE = "disable"
   ```

   **macOS/Linux (Bash)**:
   ```bash
   export PGSSLMODE=disable
   ```

3. **Execute migrations**:

   ```bash
   supabase db push --db-url "postgres://postgres:<POSTGRES_PASSWORD>@<supabase-host>:5432/postgres"
   ```

   Replace placeholders:
   - `<POSTGRES_PASSWORD>`: The PostgreSQL password configured in Step 3
   - `<supabase-host>`: The Supabase domain (e.g., `supabase.sems-maintainers.com`)

   This command executes all migration files sequentially, creating:
   - Core database tables (profiles, events, organizations, registrations)
   - Lookup tables (roles, statuses, event states)
   - Row Level Security (RLS) policies
   - Audit trail triggers and functions

4. **Verify migration success**:

   Access Supabase Studio and confirm:
   - **Table Editor**: All tables are present with correct schemas
   - **Authentication** → **Policies**: RLS policies are enabled and configured

   - Configure OAuth 2.0 credentials
   - Set authorized redirect URIs:

   - Update the Supabase environment variables (completed in Step 3)

### Configure Resend Email Service

Set up Resend for transactional email delivery, including event notifications, reminders, and certificate generation.

1. **Create a Resend account**:

   Register at [resend.com](https://resend.com) and verify the account.

2. **Configure and verify the domain**:

   - Navigate to **Domains** → **Add Domain**
   - Enter the domain name (e.g., `university.edu`)
   - Add the provided DNS records to the domain registrar:
     - SPF record
     - DKIM record
     - DMARC record
   - Wait for DNS propagation and verification (typically 5-15 minutes)

3. **Generate API key**:

   - Navigate to **API Keys** → **Create API Key**
   - Configuration:
     - Name: `SEMS2 Production`
     - Permission Level: **Send Access**
   - Copy the generated API key (format: `re_...`)
   - Store securely for use in Step 7

</Step>

## Post-Deployment Verification

After completing all deployment steps, verify the following:

1. **Application Access**: Confirm all services are accessible via their configured domains
2. **Authentication**: Test Google OAuth login flow
3. **Database Connectivity**: Verify data persistence and retrieval
4. **Email Delivery**: Send test emails to confirm Resend integration
5. **SSL Certificates**: Verify HTTPS is functioning on all domains

## Troubleshooting

Common issues and solutions:

- **Build failures**: Check Docker daemon status and available disk space
- **Database connection errors**: Verify PostgreSQL credentials and network connectivity
- **OAuth errors**: Confirm redirect URIs match exactly in Google Cloud Console
- **Email delivery failures**: Verify Resend API key and domain verification status

For additional support, consult the respective documentation for Dokploy, Supabase, and SvelteKit.
